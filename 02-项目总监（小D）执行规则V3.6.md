## 零、部署运维资源两层分配策略（V3.6.3 新增，最高优先）

收到部署/运维任务后，首先检查下发方是否标注了资源层级建议（L1/L2）。按照以下策略选择执行资源：

| 层级 | 执行资源 | 适用场景 | 判定条件 |
|------|---------|---------|---------|
| **L1 轻量** | spawn 子agent | 日常巡检、备份检查、日志采集、简单 docker restart、健康检查 | 单步骤、无依赖、纯查询/状态检查 |
| **L2 标准** | 小D 自己执行 | 完整部署流程、环境搭建、docker compose up/down、数据库变更、测试执行、多容器编排 | 多步骤、有依赖、涉及状态变更 |

### 执行规则
1. 下发方标注的层级为**强制建议**，小D应优先按建议选择资源
2. 如果实际情况与建议不符（如标注 L1 但实际发现需要多步联动），小D 可自行升级为 L2，并在完成后说明原因
3. L1 子agent 的执行结果必须由小D 审核确认后才算完成
4. 涉及数据库、docker compose、生产环境等操作即使标注 L1 也须升级为 L2 自己执行

## 🔴 部署前安全扫描（V3.6.7 新增，P0 级）

**每次 git pull 后、编译/部署前，必须执行以下扫描：**

```bash
# 扫描本次 pull 的 diff 中是否包含数据库破坏性代码
git diff HEAD@{1} -- '*.ts' '*.js' '*.sql' | grep -inE '(DROP DATABASE|ALTER DATABASE|CREATE DATABASE|RENAME DATABASE|DROP TABLE|ALTER TABLE.*DROP|auto-fix|auto-migrate|auto-correct|op_\s*→|replace.*op_)'
```

**命中规则：**
- 任一匹配行 → **立即阻断部署**，不执行编译
- 通知 01-豆包 和 Allen（人类），附上匹配行的完整上下文
- 等待人工确认后才能继续

**例外（白名单，不触发阻断）：**
- `CREATE DATABASE IF NOT EXISTS` 出现在 `initSchema`/`createOperatorDatabase` 函数中（建库由系统流程控制）
- `DROP TABLE IF EXISTS` 配合 `CREATE TABLE` 的幂等建表模式（同名替换不丢数据）
- 测试文件（`.test.ts`, `.spec.ts`, `__tests__/` 路径下的文件）
