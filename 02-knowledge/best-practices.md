## 📋 分类目录

### 体系与规则管理
1. **规则分层管理**
   - **适用场景：** 任何需要明确 AI Agent 权责的体系治理
   - **具体做法：** 五层规则体系（L1宪法→L2岗位→L3规范→L4技能→L5知识），效力从上到下递减，下位规则不得违反上位规则
   - **为什么这么做：** 层级清晰、修改范围可控、底层可灵活迭代
   - **反模式：** 把所有规则塞进一个文件，改一处影响全局

2. **岗位合并原则**
   - **适用场景：** 体系初始设计岗位过多需要精简
   - **具体做法：** 只有天然在同一治理单元内的岗位才合并（如"执行Agent+架构师+代码调度"这些都属于技术执行单元）；跨治理单元的不合并（如产品和开发、测试和运维）
   - **为什么这么做：** 避免权责混淆，保持权责分离

3. **独立监督序列不纳入 OpenClaw Agent 体系**
   - **适用场景：** 审计、合规、纪律检查类角色
   - **具体做法：** 监督者与被监督者不应运行在同一套 Agent 体系内，否则存在互相影响、互相包庇的风险
   - **为什么这么做：** 真正的独立才能保证审计的有效性

### Agent 间同步
1. **跨 Agent 信息同步走会话消息机制**
   - **适用场景：** 多个 OpenClaw Agent 实例之间需要同步信息
   - **具体做法：** 使用 `sessions_send` 发送消息，不直接编辑对方文件；接收方必须先有活跃 session 才能收到消息
   - **为什么这么做：** 安全权限隔离，每个 Agent 只操作自己的文件
   - **反模式：** 通过 file_system MCP 直接写入对方 agent 目录的文件

### 安全与权限
1. **密钥网关三级鉴权**
   - **适用场景：** 体系中多个 Agent 共享密钥但需要严格权限管控
   - **具体做法：** 黑名单拦截（特定 Agent 永久禁止获取模型密钥）→ 白名单校验（仅特定 Agent 可解密敏感密钥）→ vault 权限校验
   - **为什么这么做：** 纵深防御，单层绕过还有多层拦截
   - **反模式：** 黑名单不允许出现在白名单之前（应先禁再放）

2. **操作前必须执行安全检查清单**
   - **适用场景：** 任何写/删/改/破坏性操作
   - **具体做法：** 打开 `安全操作检查清单.md` 逐项自检，全部通过才能执行
   - **为什么这么做：** 防呆机制，减少操作失误概率
   - **反模式：** 凭经验跳过检查

### 模型调度
1. **默认低成本模型，升级需审批**
   - **适用场景：** 体系内所有 AI Agent 的模型使用
   - **具体做法：** 日常业务默认使用低成本模型（如 DeepSeek V3）；核心复杂模块使用高级模型但需审计；禁止无审批使用 high-cost 模型
   - **为什么这么做：** 成本可控原则，审计事后追溯
   - **反模式：** 所有任务都用最强模型

### 数据库操作
1. **高危操作必须请示人类**
   - **适用场景：** `DELETE`/`DROP`/`ALTER`/`TRUNCATE`/批量 `UPDATE`
   - **具体做法：** 必须先问 Allen，得到明确批准才能执行；Allen 不回复则继续等，不能自己动手
   - **为什么这么做：** 测试数据删了事小，生产用户数据和资金数据删了无法挽回
   - **允许不请示的操作：** `SELECT`、业务正常 `INSERT`、单条 `UPDATE`

### 开发工作流
1. **新任务开始前必须做版本对齐检查**
   - **适用场景：** 每个新开发任务开始前
   - **具体做法：** `git status` 检查本地有无未提交改动 → 三端（本地/GitHub/main/服务器）commit 对比 → 确认一致或人类判定无影响后才开发
   - **为什么这么做：** 防止在旧代码基础上开发导致冲突或覆盖
   - **反模式：** 不看 git 状态直接开工

2. **原文输入，原文写入**
   - **适用场景：** 配置文件中涉及 Key、密码、URL、手机号等敏感/精确字符串
   - **具体做法：** Allen 发来的原始字符串禁止做任何格式修改（不能加省略号、不能 truncate、不能格式化）
   - **为什么这么做：** 确保配置精确无误
