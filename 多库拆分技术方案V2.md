## 一、背景与目标

### 1.1 业务诉求
每个运营商需要独立的数据库，实现数据物理隔离。不额外租用服务器，在同一台服务器上通过 MySQL 库级隔离实现。

### 1.2 核心目标
- **数据隔离**：A 运营商无法访问 B 运营商数据
- **成本可控**：共用一台服务器，不分租额外资源
- **渐进演进**：不一次性全量拆分，分三阶段平滑过渡
- **不影响线上**：核心交易链路（订单/支付）在当前阶段不物理拆分

## 二、核心架构选型

### 2.1 模式：Database-per-Tenant
```
MySQL 实例 (127.0.0.1:3308)
├── robot_maze_race              ← 主库（超管、全局配置）
├── robot_maze_race_op_001       ← 运营商A独立库
├── robot_maze_race_op_002       ← 运营商B独立库
└── robot_maze_race_summary      ← 总部汇总库（阶段2）
```

### 2.2 拆分边界

#### 主库保留（不拆分）
| 表名 | 理由 |
|------|------|
| `operators` | 总部管理的运营商元数据 |
| `operator_members` | 运营商成员账号，登录态唯一入口 |
| `admin_users` / `admin_roles` | 超管后台 |
| `settings` | 全局配置 |
| `banks` | 银行字典，公共数据 |
| `seasons` | 赛季，全局配置 |

#### 按运营商拆分（迁入独立库）— 跟运营商走的全部业务数据
| 表名 | 业务域 | 说明 |
|------|--------|------|
| `venues` | 场地管理 | 物理地点，运营商下有多个场地 |
| `races` | 比赛 | 一场比赛的配置（赛制、规则、奖品） |
| `race_rooms` | 赛房 | 比赛内的分组房间，层级：运营商→场地→比赛→赛房 |
| `race_packages` | 参赛包 | 运营商自定义的参赛套餐 |
| `referees` | 裁判管理 | 运营商招募的裁判 |
| `referee_invites` | 裁判邀请 | 裁判邀请链接与注册 |
| `users` | 用户 | 首次扫码注册锁定 operator_id，不可自行切换 |
| `user_profiles` | 用户资料 | 头像、昵称、等级等 |
| `merchants` | 商家 | 运营商线下发展的商家，独立不通用 |
| `orders` | 订单（明细账） | 订单按 operator_id 分发到各运营商库 |
| `points_transactions` | 积分流水 | 用户积分，只在本运营商内有效 |
| `points_exchange_log` | 积分兑换记录 | 同上 |
| `user_coupons` | 用户优惠券 | 商家发的券只在本运营商内有效 |
| `merchant_coupons` | 商家优惠券模板 | 同上 |
| `expand_coupons` | 扩展优惠券 | 同上 |
| `help_requests` | 助力请求 | 社交助力 |
| `help_helpers` | 助力记录 | 助力明细 |

#### 放总部主库 — 跨运营商共享与管控数据
| 表名 | 业务域 | 说明 |
|------|--------|------|
| `operators` | 运营商管理 | 总部管理所有运营商的元数据 |
| `operator_members` | 运营商成员 | 登录态唯一入口，含 operator_id 外键 |
| `admin_users` | 超管用户 | 总部后台 |
| `admin_roles` | 超管角色 | 权限控制 |
| `settings` | 全局配置 | 跨运营商共享配置 |
| `banks` | 银行字典 | 公共数据 |
| `seasons` | 赛季 | 全局赛季配置 |
| `regions` | 省市区 | 公共地理数据 |

## 三、全场景租户识别机制

| 场景 | 入口 | 识别方式 | 兜底 |
|------|------|---------|------|
| **运营商后台登录** | `/operator/login` | JWT token 解析 `operator_id` | 403 |
| **超管后台** | `/admin/*` | `role=admin`，切换运营商视图 | 汇总库查询 |
| **前端小程序** | 统一入口 | 用户归属的 `operator_id` 自动路由 | 403 |
| **支付回调** | 总部支付网关 | 订单号反查 `operator_id` → 分发写入 | retry + 对账 |
| **裁判注册** | `/referee/invite/:token` | token 解析 `operator_id`（referee_invites 表） | 无效 token → 410 |
| **API / Webhook** | 所有业务接口 | 中间件强制校验 `ctx.state.operatorId` | 403 |

### 运营商标识来源（优先级）
1. JWT token 中的 `operator_id`（已登录用户）
2. 邀请 token 中的 `operator_id`（裁判注册）
3. 用户首次扫码注册时锁定的 `operator_id`（用户→运营商绑定）
4. URL 参数中的 `operator_id`
5. **无以上任一标识 → 直接 403**

### 数据归属核心规则
- **用户归属**：首次在哪个运营商小程序二维码注册 = 属于哪个运营商，不可自行切换（需线下申请）
- **商家归属**：运营商线下发展的商家，商家发行的券只在运营商内部有效
- **积分/等级/券/抵扣卡**：全部跟着用户走 → 跟着运营商走，互不通用

### operator_members 路由逻辑
`operator_members` 表保留在主库，包含 `operator_id` 外键。登录后 JWT 颁发时写入 `operator_id`，后续所有业务请求通过 token 中的 `operator_id` 路由到对应独立库。**一个成员只属于一个运营商，不存在跨库路由歧义。**

## 四、全局唯一 ID 方案

### 4.1 当前问题
多库独立后，各库自增主键（INT AUTO_INCREMENT）会产生重复 ID，导致跨库数据合并时订单号/用户ID冲突。

### 4.2 方案：雪花算法 + 租户号段
```
ID 结构（64位）：
┌─┬─────────────────────────┬───────────┬──────────┐
│1│     时间戳 (41位)        │ 机器ID(10位)│序号(12位) │
└─┴─────────────────────────┴───────────┴──────────┘

机器ID分配：
  bit 0-5：运营商编号（0-63，主库用 0）
  bit 6-9：实例编号（支持多实例部署）
```

### 4.3 实施节奏
- **阶段1（当前）**：不实施。单库运行，无需全局 ID。
- **阶段2**：引入雪花算法，改造 `users.id`、`orders.id`、`points_transactions.id` 为主键生成方式。

## 五、总部管控分层架构

```
┌──────────────────────────────────────────────┐
│              超管后台 /admin                  │
│                                              │
│  ┌─────────────┐  ┌──────────────────────┐   │
│  │ 主库 (配置)  │  │ 总部汇总库 (统计报表)  │   │
│  │ operators    │  │ - 订单/支付概览       │   │
│  │ settings     │  │ - 用户增长           │   │
│  │ accounts     │  │ - 积分/卡券大盘       │   │
│  └─────────────┘  └──────────────────────┘   │
│                          ↑                    │
│                  定时同步 (每小时)              │
│         ┌────────┬────────┬────────┐         │
│         │ op_A库  │ op_B库  │ op_C库 │  ...   │
│         └────────┴────────┴────────┘         │
└──────────────────────────────────────────────┘
```

### 总部三层权限
| 层级 | 权限 | 数据库访问 |
|------|------|----------|
| **超级管理员** | 全平台排查 | 可跨库精准查询（仅排查，留审计日志） |
| **运营管理员** | 报表/对账 | 仅汇总库 |
| **运营商管理员** | 本运营商业务 | 仅本运营商库 |

**禁止行为**：全表遍历跨库查询、实时大盘跨库 JOIN

## 六、统一收银与资金分账

### 6.1 架构
```
用户付款 → 总部统一下单接口 → 微信/支付宝
         → 支付成功回调
           ├→ 写入主库 payments（总账）
           ├→ 按 operator_id 分发订单到运营商库（明细账）
           └→ retry 3次确保分发成功
         
清分：第三方合规空中清分系统 → 直接打款给运营商
      → 回调写入主库 settlement_records（清分记录）
```

### 6.2 数据存放规则
| 数据 | 位置 | 用途 |
|------|------|------|
| 支付流水（总账） | 主库 `payments` | 总部记账、财务报表 |
| 订单明细（明细账） | 各运营商独立库 `orders` | 运营商查看自己的订单 |
| 清分记录 | 主库 `settlement_records` | 第三方清分系统回调，记录打款流水 |

### 6.3 对账机制
- **每日自动对账**：主库 payments 总额 ↔ 各运营商库 orders 明细汇总
- **清分对账**：settlement_records 打款金额 ↔ 对应运营商 orders 应收金额
- **不一致告警**：飞书通知 + 人工介入，确保一分不差

## 八、备份与灾备

### 8.1 备份策略
- 每个运营商库独立备份（`mysqldump --single-transaction`）
- 每日凌晨全量备份 + 每小时 binlog 增量
- 保留周期：近7天每日全量 + 近30天每周全量 + 近90天每月全量

### 8.2 单库恢复
- 支持按时间点恢复单个运营商库，不影响其他库
- 测试环境先验证 → 正式环境恢复

### 8.3 回收站机制
- 关键表（users, orders, races）增加 `deleted_at` 软删除字段
- 删除操作先标记 `deleted_at = NOW()`，7天后定时任务物理清理
- 误删恢复：通过 `deleted_at IS NOT NULL` 反查

## 十二、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 连接数膨胀 | MySQL 拒绝连接 | 动态连接池（HikariCP），中小库2-5连接，大客户10-20连接 |
| Schema 不同步 | 独立库表结构不一致 | Flyway 版本管理，每日巡检一致性 |
| 跨库查询性能 | 超管报表慢 | 90%场景走汇总库，10%低频精准查询 |
| 单库宕机 | 该运营商停服 | 故障隔离，其他运营商不受影响 |
| 数据迁移差错 | 数据丢失/错乱 | 先备份再迁移，迁移后比对 checksum |
