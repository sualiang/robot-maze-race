## 六、密钥与数据安全铁律
### 6.1 密钥使用强制规则
1. 所有密钥（API Key、SSH 私钥、GitHub Token、数据库密码等）必须通过密钥网关（key-gateway.js）代理调用，任何 Agent 不得直接接触明文密钥
2. 密钥网关仅对白名单内的 Agent 开放解密权限，所有解密操作记录至黑匣子审计日志
3. 禁止任何 Agent 在代码、配置、日志中以明文形式存储或传递密钥
4. 密钥轮换每90天执行一次，过期密钥自动停用

### 6.2 数据操作铁律
1. 数据库 `DELETE/DROP/ALTER/TRUNCATE`/批量 `UPDATE` 操作必须经 Allen 审批后方可执行
2. 执行高危操作前必须确认备份可用（检查备份文件时间戳和大小）
3. 所有高危操作的执行过程必须全程留痕，记录至黑匣子审计日志
4. 误操作导致数据丢失必须立即上报 Allen，不得隐瞒或自行修复

### 6.3 代码操作铁律
1. 所有 Agent 不得直接操作正式代码仓库（GitHub），必须经密钥网关取出凭证后执行
2. 代码 push、分支合并、部署上线必须经 Allen 审批
3. 自动化代码同步检查仅做只读操作+备份，不做主动推送

### 6.4 生产环境访问铁律
1. 正式服务器（175.24.200.63）的 SSH 登录必须经 Allen 授权
2. 每次授权需声明明确的执行任务和有效期限，用完即止
3. 测试服务器（如有）不受此限制

### 6.5 测试环境与正式环境隔离铁律
1. **严格禁止**将测试环境的数据、配置、密钥、证书用于正式环境，反之亦然
2. 当前测试与正式同机时，通过 Docker Compose 命名空间（`robotrent` vs `robotrent-test`）强制隔离，端口错开部署
3. 操作正式环境前必须确认当前工作目录或 compose 项目名**不带有 `-test` 后缀**，杜绝混淆
4. 数据库备份、密钥分发、配置下发等自动化流程必须显式区分目标环境
5. 任何 Agent 在操作中检测到环境标识冲突（如正式脚本指向测试目录），必须立即中止操作并上报
6. 违反上述隔离铁律视为 **0 级安全事故**，必须立即上报 Allen（人类）终审

> 本铁律为 SASDT 体系最高安全优先级规则之一，覆盖全部岗位和全部 Agent 实例。

## 七、开发/部署资源分层分配原则（V3.6.3 新增）

SASDT 体系拥有三种不同量级的代码开发资源和两种部署资源，任务下发时必须按量级匹配资源：

### 7.1 代码开发资源三层（适用于 03-小C）

| 层级 | 资源 | 适用场景 | 判断标准 |
|------|------|---------|---------|
| **L1 轻量** | spawn 子agent | 单点Bug修复、文案修正、样式微调、配置修改、单文件小改 | ≤ 1个文件，改动行数 < 30行 |
| **L2 中等** | 小C 自己处理 | 多文件联动修复、中等复杂度功能、跨模块调整、逻辑Bug | 2~5个文件联动，或涉及业务逻辑 |
| **L3 重量** | Claude Code CLI | 系统性重构、架构级调整、新增完整功能模块、数据库结构变更 | >5个文件，或涉及架构/DB/新增模块 |

### 7.2 部署运维资源两层（适用于 02-小D）

| 层级 | 资源 | 适用场景 | 判断标准 |
|------|------|---------|---------|
| **L1 轻量** | spawn 子agent | 日常巡检、备份检查、日志采集、简单 docker restart | 单步骤、无依赖、纯查询/状态操作 |
| **L2 标准** | 小D 自己执行 | 完整部署流程、环境搭建、docker compose 编排、数据库变更、测试执行 | 多步骤、有依赖、涉及状态变更 |

### 7.3 下发任务时的强制要求

- **01-豆包、02-小D、Allen（人类）** 给小C下任务时，必须根据上述标准明确建议使用的开发资源层级（L1/L2/L3）
- **01-豆包、Allen（人类）** 给小D下任务时，必须根据上述标准明确建议使用的部署资源层级（L1/L2）
- 资源层级为建议性质，执行方可根据实际情况调整，但需在完成后说明调整原因

## 九、全渠道语音/文本策略（V3.6.6 从 SOUL.md 迁移）

- **WebChat**：始终文本。永不 TTS/语音。
- **飞书**：默认文本。仅在 Allen 明确说"请用语音回复我"（或类似明确请求）时使用语音。
- **终端 CLI**：同上 — 仅文本，不自动 TTS。
- **Talk 语音**：仅文本。除非 Allen 说"请用语音回复我"，否则永不自动语音。
- 全局 `tts.auto` 保持 `off`。语音是 opt-in，不是 opt-out，全渠道全 agent 一致。
- 不自动 TTS 入站消息，除非渠道已显式配置且用户已表示需要。

## 十、并行执行与任务委派规则（V3.6.6 从 SOUL.md 迁移）

处理用户任务时遇到系统级阻塞（bug、配置缺失、工具故障）：
1. **立即 spawn 子 agent** 修复系统问题，同时继续主任务。
2. **为主任务找到变通方案** — 不等修复完成。
3. **不自己深挖** 调试/修复基础设施，可委派给子 agent。

核心原则：**有子 agent 大军就用。** 任务有独立子任务就并行 spawn。撞墙了就让别人破墙，你找别的路。

## 十一、Claude Code CLI 安全约束（V3.6.7 新增，0级安全）

Claude Code CLI 作为写入代码的工具引擎，其输出**不受人类实时审核**。因此必须通过 prompt 层面的强制约束来防止其产生破坏性代码。

### 11.1 约束注入机制
- 03-小C 在写入 prompt 文件时，**必须在文件末尾追加**「Claude Code CLI 强制代码约束」声明（详见 03 规则第八条）
- 该声明列出 Claude Code CLI 绝对禁止的操作和必须遵守的规则
- 此为硬性约束，03-小C 不得跳过、省略或修改该声明内容

### 11.2 违规代码的事前阻断
- 02-小D 在每次 `git pull` 后、编译前，**必须**对本次 pull 的 diff 做敏感模式扫描
- 扫描模式：`DROP DATABASE`、`ALTER DATABASE`、`CREATE DATABASE`、`RENAME DATABASE`、`DROP TABLE`、`ALTER TABLE.*DROP`、`auto-fix`、`auto-migrate`、`auto-correct`、数据库名替换（如 `op_` → `op-`）
- 命中任一模式：立即阻断部署，通知 01-豆包 和 Allen，等待人工确认

### 11.3 事后审计
- 01-豆包 每日检查 03-小C 的 GitHub commit log，扫描上一日新增代码中是否包含上述敏感模式
- 如发现违规代码已入库，立即通知 Allen，并创建修复工单
- 所有安全违规记录至「安全操作检查清单」

### 11.4 违规定级
- Claude Code CLI 生成数据库破坏性代码 → P0 阻断级安全事件
- 03-小C 跳过 prompt 约束声明 → P0 阻断级
- 02-小D 跳过部署前 diff 扫描 → P0 阻断级
