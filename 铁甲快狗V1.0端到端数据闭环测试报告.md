---

## 一、测试方案概述

### 1.1 测试目标
验证「玩家购买参赛包」完整商业链路的前后端数据闭环正确性：
```
前端操作 → API调用 → 业务逻辑执行 → 数据库写入 → 跨表数据一致性
```

### 1.2 测试策略
- **API 测试层:** 通过 supertest 直连 Express app 发送 HTTP 请求，验证返回状态码 + 响应结构
- **数据库验证层:** SQLite 直连查询，每步 API 调用后 SELECT 校验数据库写入结果
- **数据隔离:** `DB_FORCE_SQLITE=true` 强制使用 SQLite 内存数据库，零污染生产 MySQL
- **数据清理:** 每个测试套件 `afterAll` 自动清理测试数据

### 1.3 测试链路
本次覆盖 **1 条核心链路**（玩家购买参赛包），包含 2 个测试套件、**25 条断言**：

| 链路编号 | 链路名称 | 测试套件 | 断言数 | 覆盖端 |
|---------|---------|---------|-------|--------|
| L1 | 玩家购买参赛包 → 抵扣金扣减 → 参赛次数增加 → 成长值发放 | API + DB | 25 | 玩家端 |

## 五、端到端数据闭环验证结论

### 5.1 全链路数据流追踪（L1 链路）

```
┌──────────────────────────────────────────────────────────────┐
│  ① 玩家登录获取 token                                         │
│  ↓                                                           │
│  ② GET /player/packages → 返回参赛包列表                      │
│  ↓                                                           │
│  ③ POST /player/orders  { packageId, useDeduction }          │
│     ├─ 订单写入 orders 表                    ✅ 金额正确      │
│     ├─ 抵扣金扣减 entry_deductions (status→used) ✅ 面额一致  │
│     ├─ 参赛次数增加 users.race_count          ✅ 次数对齐     │
│     ├─ 成长值发放 orders.remaining_growth     ✅ 与包一致     │
│     └─ 赠送抵扣金 entry_deductions (INSERT)   ✅ 到期365天    │
│  ↓                                                           │
│  ④ GET /player/orders → 订单状态回读 ✅                      │
│  ⑤ GET /me/profile-check → 参赛次数回读 ✅                    │
│  ⑥ GET /player/deductions → 抵扣金回读 ✅                     │
│                                                              │
│  端到端：写入 4 张表 → 回读 3 个 API → 100% 数据一致 ✅       │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 关键结论

| 数据节点 | 写入验证 | 回读验证 | 跨表一致性 |
|---------|---------|---------|-----------|
| 订单金额 | ✅ | ✅ | orders↔entry_deductions ↔ 100% |
| 参赛次数 | ✅ | ✅ | orders↔users↔API ↔ 100% |
| 抵扣金 | ✅ | ✅ | entry_deductions↔orders↔API ↔ 100% |
| 成长值 | ✅ | ✅ | orders↔race_packages ↔ 100% |
| 安全拦截 | ✅ 401×2 | N/A | 越权 100% 拦截 |
