机器狗迷宫竞速赛事小程序 — AI 开发小组实施方案
文档版本：V1.0 | 制定日期：2026-06-04 | 状态：待审核

一、项目概述
本项目为"机器狗迷宫竞速赛事"全平台系统，覆盖 5 个终端：玩家微信小程序、裁判微信小程序、运营商小程序+网页后台、赛场大屏网页、总部管理后台。为确保高效交付，采用 AI 开发小组模式，由 1 名架构师 + 多名 AI Agent 并行开发。

三、模块分解与任务分配
3.1 🐘 后端核心 Agent（DB+API）
负责所有后端服务，是整个系统的支柱。
工作内容：
数据库 Schema 设计：用户表、赛场表、裁判表、参赛包表、订单表、签到记录表、比赛成绩表、助力记录表、膨胀券表、考勤记录表等（约 15+ 张表）
RESTful API 开发：约 40-50 个接口
  • 用户模块：微信登录、个人信息 CRUD、注销
  • 赛场模块：CRUD、状态管理、签到二维码生成
  • 裁判模块：认证、绑定赛场、签到/签退
  • 参赛包模块：CRUD、上下架
  • 订单支付模块：下单、支付回调、退款
  • 比赛模块：排队队列、成绩记录
  • 榜单模块：日/月/年榜单计算
  • 助力模块：发起、助力、风控校验
  • 膨胀券模块：发放、核销、有效期管理
  • 财务模块：结算、提现、分润计算
  • 考勤模块：记录查询、导出
WebSocket 服务：排队叫号、秒表计时同步、成绩实时推送
聚合支付 SDK 对接（微信/支付宝/银联）
Redis 排队队列实现
断网数据同步逻辑
预估工作量：5-6 天 ☕☕☕☕☕☕
3.2 📱 玩家端小程序 Agent
负责参赛玩家体验的全链路小程序开发。
工作内容：
微信小程序项目初始化 + 全局配置
微信一键登录与自动注册
首页：赛事介绍、参赛包购买入口、榜单入口
参赛包购买页：列表展示、微信支付下单
签到排队流程：扫码、赛场状态校验、参赛次数校验、首次信息强制填写、排队入列
榜单页：日/月/年三级榜单、高亮自己排名
个人中心：个人信息编辑、参赛记录、助力记录、优惠券列表、参赛次数管理
助力裂变：分享卡片生成、助力状态展示
断网本地缓存逻辑
预估工作量：4-5 天 ☕☕☕☕☕
3.3 📱 裁判端小程序 Agent
负责现场裁判核心操作工具。
工作内容：
微信小程序项目初始化
微信一键登录 + 待认证状态处理
签到/签退管理页：GPS 定位获取、签到激活赛场、签退暂停赛场、考勤记录展示
主操作页：排队列表、当前比赛选手高亮
秒表控制：开始比赛、结束比赛、毫秒计时实时同步
超时判罚：自动记录最大超时时间
异常中断：机器狗故障处理（保留参赛次数）
断网本地缓存 + 联网自动同步
大屏登录二维码扫码激活（赛场大屏端联动）
预估工作量：3-4 天 ☕☕☕☕
3.4 🌐 前端网页 Agent（运营商后台 + 大屏 + 总部后台）
一人负责三个网页端，因为三者共享组件和设计模式。
工作内容：
A. 运营商网页后台
赛场管理：新建（含二维码下载打印）、编辑、停用、状态监控
裁判管理：认证列表、认证/禁用操作、绑定赛场、考勤查询导出
参赛包管理：CRUD、上下架
营销管理：助力活动开关+奖励设置、膨胀券活动开关+赠送设置
财务中心：营收明细、结算明细、T+1提现、成本记账
B. 赛场大屏展示端
专属网址 + 登录二维码页（裁判扫码激活）
大屏模式：赛场名称+状态、实时计时（毫秒）、排队叫号、前10排行榜
WebSocket 实时自动更新
全屏适配、断网"网络连接异常"提示
C. 总部管理后台
运营商管理：创建/编辑/禁用、详情查看
全局考勤管理：全平台筛选+导出
营销全局配置：助力人数要求、活动有效期、膨胀券有效期
财务中心：全平台统计、分润配置、支付手续费配置、提现审核、报表导出
系统设置：全局参数、风控管理、黑名单
预估工作量：5-6 天 ☕☕☕☕☕☕
3.5 📱 运营商小程序端 Agent
负责运营商老板端小程序——轻量数据看板。
工作内容：
微信小程序项目初始化
顶部汇总数据：今日总参赛人数、总营收、总预估佣金
赛场列表：名称、今日数据、运营状态
单赛场详情页：实时数据、历史趋势图、当日裁判考勤
纯展示，无操作类功能
预估工作量：1.5-2 天 ☕☕
3.6 🧪 测试+集成 Agent
负责全链路质量保障和交付。
工作内容：
编写 API 接口自动化测试（约 50+ 用例）
小程序前端模拟测试（玩家端+裁判端）
web 端功能测试（运营商后台+总部后台+大屏）
WebSocket 实时同步场景测试
断网容错场景专项测试
边界场景测试：多次签到、排队溢出、助力风控、成绩不可篡改验证
集成部署：GitHub Actions CI/CD 配置、自动部署脚本
回归测试 + 最终验收
预估工作量：持续集成，总约 3-4 天 ☕☕☕☕

4.2 依赖关系
关键依赖链：
第0步：架构师定义 API 接口规范（Swagger/OpenAPI），所有前端Agent按此规范开发
第1步：后端 Agent 优先开发登录/用户/赛场/裁判基础接口，让前端可联调
第2步：玩家端、裁判端、网页端可并行开发
第3步：运营商小程序依赖API + 数据看板设计，可滞后1-2天启动
第4步：测试Agent在第5天介入，随开发推进逐步覆盖
五、开发与协作规范
5.1 仓库结构
统一 monorepo（单一仓库管理所有代码）
packages/server/ — 后端服务
packages/mini-program-player/ — 玩家端小程序
packages/mini-program-referee/ — 裁判端小程序
packages/mini-program-operator/ — 运营商小程序
packages/web/ — 网页端（运营商后台 + 总部后台 + 大屏，三个路由）
packages/shared/ — 共享类型定义、常量、工具函数
docs/ — API 文档、数据库文档
scripts/ — 部署脚本
5.2 协作流程
所有 Agent 通过 fork 工作流提交 PR
架构师（小D）每日 code review + merge
API 规范先行：后端 Agent 先输出 OpenAPI 文档，前端按文档开发
每日站会（自动摘要）：各 Agent 汇报进度、阻塞项
所有代码提交到 GitHub mono-repo，CI 自动跑 lint + test
5.3 接口规范
统一 RESTful 风格，统一错误码格式
使用 TypeScript 类型共享 package（packages/shared）
WebSocket 消息格式统一 JSON，约定 event type
API 版本号：/api/v1/...
