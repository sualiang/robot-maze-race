## 一、缺陷分级标准

所有质量问题按严重程度分为 5 级，**全角色统一执行**：

| 等级 | 名称 | 定义 | 典型例子 | 处理要求 |
|------|------|------|----------|----------|
| **P0** | Blocker（阻塞级） | 系统崩溃、核心功能完全不可用、数据丢失、安全漏洞 | 系统崩溃、核心流程走不通、数据损坏、SQL注入、XSS、密钥硬编码 | 🔴 立即修复，不修复不能合并/上线，修复后必须验证 |
| **P1** | Critical（严重级） | 重要功能不可用、主要业务流程受影响，但有 workaround | 重要功能 Bug、接口响应 >5s、高并发不稳定、中等安全漏洞 | 🟠 必须修复，上线前必须修复或明确修复计划 |
| **P2** | Major（重要级） | 一般功能问题、用户体验差、代码可维护性差，不影响核心功能 | 次要功能 Bug、页面样式问题、代码重复、命名不规范 | 🟡 建议修复，上线前最好修复，来不及可后续迭代但必须记录 |
| **P3** | Minor（次要级） | 小问题、优化建议、代码风格问题 | 代码格式不统一、拼写错误、多余 import | 🟢 建议优化，不改也可以，不阻塞合并和上线 |
| **P4** | Info（提示级） | 建议性信息，非问题 | 优雅写法建议、组件复用提醒、最佳实践提示 | 🔵 了解即可，不纳入门禁 |

### 特殊业务模块分级参考

以下为常见高风险业务场景的默认定级参考，每个项目可在此基础上调整：

| 场景类型 | 默认定级 | 说明 |
|---------|---------|------|
| 结算金额计算错误 | P0 | 涉及资金结算 |
| 资金账目前后不一致 | P1 | 涉及用户资金权益 |
| 核心业务流程走不通 | P1 | 影响主流程 |
| 页面展示金额多了一个零 | P1 | 影响用户信任 |

> 项目特定的高风险业务场景分级，由产品总负责人（豆包API）在该项目上下文中补充。

### 3.3 测试执行关键规则（2026-07-04 实践总结）

#### 3.3.1 弹窗类型识别（铁甲快狗·总部后台）
页面弹窗类型**不假设**，必须先 debug 看真实 DOM class：
- 新建/编辑 → **Modal**（`.ant-modal`）
- 详情 → **Drawer**（`.ant-drawer`）
- 禁用/删除/重置密码 → **Popover**（`.ant-popover`，含 `.ant-popconfirm`）

#### 3.3.2 Ⅰ类控件（有API交互）必走9步
引擎启动 → 数据隔离（DB基线）→ 点击触发 → 全量Network/Console监听 → **强制确认**（点确认按钮，不可点取消跑路）→ 等待响应 → 刷新页面 → DB后置校验 → 真值比对

#### 3.3.3 Ⅱ类控件（无API交互）走5步
前置状态确认 → 触发操作 → 视觉校验 → 可逆性验证（点取消恢复原状）→ 运行时校验

#### 3.3.4 禁止事项（META_1.3 / SUB_1.1）
- ❌ 禁止纯DOM表层校验（必须验证Network请求和DB变更）
- ❌ 禁止Ⅰ类控件流程截断只点取消

#### 3.3.5 新建/编辑静默失败特征
Console `[pageerror] Object`（空异常，无stack trace）+ Network 0次请求 + Modal不关闭 + 无Toast = 表单提交函数异常被 catch 吞掉。此类缺陷无法通过 DOM 校验发现，必须靠 Network 监听捕获。

## 八、裁判签到架构（Phase 1+2）

### 8.1 签到方式（已移除 GPS）
- ✅ QR 码手动输入 6 位激活码 → `POST /attack/check-in-by-qr`
- ❌ GPS/地理编码/Haversine 距离计算（Phase 1 全部移除）
- ❌ 浏览器定位 `navigator.geolocation`（已删除）

### 8.2 QR 签到激活大屏数据流
```
大屏 ScreenLogin → 生成6位码(60s TTL) → WS screen_login → 后端存映射
裁判输入码 → POST /attendance/check-in-by-qr → 查映射 → 写attendance → WS推activated
大屏收到activated → 2s后跳转ScreenDisplay
```

### 8.3 激活码生命周期
- 6 位字母数字，60s TTL
- 存储在 `ws/handler.ts` 的 `Map<string, { ws, createdAt }>` 中
- 过期自动回收，断连自动清理

## 九、数据库约束陷阱

### 9.1 users.role 字段
- **类型**: `NOT NULL DEFAULT 'player'`
- **禁止**: 任何 `SET role = NULL` 操作会导致 MySQL error
- **正确**: 删除裁判时 `SET role = 'player'`（回归默认角色），不删除 user 本身

### 9.2 OAuth 回调创建记录陷阱
- OAuth 回调（`auth.ts` mp-oauth）不应**自动创建** referee pending 记录
- 正确做法：只返回 `is_new_user` 标记，创建记录仅在 `POST /referee/register` 表单提交时执行
- 违例后果：OAuth 创建的 pending 记录被注册查重匹配到 → 提示"已注册"但实际为新用户
